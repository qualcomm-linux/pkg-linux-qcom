# linux-image-qcom — Debian/Ubuntu Kernel Packaging

This repository provides the `debian/` packaging tree for building and
distributing the Qualcomm ARM64 Linux kernel as standard Debian/Ubuntu packages.

---

## Package naming

Packages follow the standard Debian/Ubuntu kernel naming convention:

| Package | Role | Example |
|---------|------|---------|
| `linux-image-<KVER>-qcom` | Kernel image, modules, DTBs | `linux-image-6.12.0-qcom-next-20260210-qcom` |
| `linux-headers-<KVER>-qcom` | Headers for out-of-tree modules | `linux-headers-6.12.0-qcom-next-20260210-qcom` |
| `linux-image-<KVER>-qcom-dbg` | Debug symbols | `linux-image-6.12.0-qcom-next-20260210-qcom-dbg` |

**`<KVER>`** is the full `kernelrelease` string (`uname -r`), which includes the
base kernel version and the LOCALVERSION suffix encoding the branch and snapshot
date (e.g., `-qcom-next-20260210`).

**`-qcom`** is a static flavour suffix appended by the packaging, identifying
Qualcomm-packaged kernels independently of the branch name.

All three packages `Provides: linux-image` / `linux-headers` / `linux-image-dbg`
virtual packages, making them interchangeable with standard distribution kernels
for dependency resolution.

## Build flow

### Overview

```
┌─────────────────────────────────────────────────────────────────┐
│  build-kernel.sh (or CI equivalent)                             │
│                                                                 │
│  1. Set up kernel source (clone / checkout branch or tag)       │
│  2. Copy debian/ from pkg-linux-kernel into kernel source root  │
│  3. [Optional] Copy selected fragments from config-available/   │
│     into config/ to activate them for this build                │
│  4. Determine KVER via make kernelrelease (kernel-source step)  │
│                                                                 │
│  5. make -f debian/rules prepare KVER=$KVER  ◄── handoff        │
│         ↓ generates debian/control                              │
│         ↓ generates debian/changelog                            │
│                                                                 │
│  6. dpkg-buildpackage / sbuild / docker_deb_build.py            │
│         ↓ reads debian/control (real package names)             │
│         ↓ calls debian/rules build / install / etc.             │
│         ↓ produces .deb files                                   │
└─────────────────────────────────────────────────────────────────┘
```

Steps 1–4 are kernel-source concerns handled by the build orchestrator.
Step 5 is the single handoff point into the packaging infrastructure.
Step 6 is a standard, unmodified Debian build tool invocation.

### The `prepare` target

`debian/rules prepare` is the packaging entry point. It substitutes `@KVER@`
in the `.in` templates to produce the final `debian/control` and
`debian/changelog` that `dpkg-buildpackage` requires.

```bash
# From the kernel source root (after copying debian/ into it):
make -f debian/rules prepare KVER=6.12.0-qcom-next-20260210
```

This produces:
- `debian/control` — with package names `linux-image-6.12.0-qcom-next-20260210-qcom`, etc.
- `debian/changelog` — with source package `linux-image-6.12.0-qcom-next-20260210-qcom (1-1)`

**Why this step is required:** `dpkg-buildpackage` reads `debian/control` before
calling any `debian/rules` targets. The `@KVER@` substitution must therefore
happen before `dpkg-buildpackage` is invoked. This is the same pattern used by
Debian's official `linux` source package and Ubuntu OEM kernels.

### LOCALVERSION and KVER

`KVER` is the full `kernelrelease` string (`uname -r`). It is composed of:

```
KVER = <base-version> + LOCALVERSION
     = 6.12.0          + -qcom-next-20260210
     = 6.12.0-qcom-next-20260210
```

`build-kernel.sh` determines KVER by:
1. Running `make defconfig` to create `.config`
2. Disabling `CONFIG_LOCALVERSION_AUTO` (prevents git hash appended to version)
3. Running `make -s kernelrelease LOCALVERSION=-<branch>-<abi>`

The LOCALVERSION suffix encodes the branch name and snapshot/ABI date. For
tagged builds, `build-kernel.sh` auto-extracts this from the tag name.

`debian/rules` recovers LOCALVERSION from the package name at build time by
stripping the `linux-image-` prefix and the trailing `-qcom` flavour suffix:

```
linux-image-6.12.0-qcom-next-20260210-qcom
→ strip "linux-image-6.12.0" → -qcom-next-20260210-qcom
→ strip "-qcom" suffix       → -qcom-next-20260210   (= LOCALVERSION)
```

## Build instructions

### Prerequisites

```bash
sudo apt-get install -y \
  build-essential devscripts debhelper-compat bc bison flex libssl-dev \
  libelf-dev dwarves python3 kmod cpio rsync pkg-config make
```

### Using build-kernel.sh (recommended)

`build-kernel.sh` is included in this repo and orchestrates the full workflow:
clone → prepare → build. Run it from the repo root.

```bash
# Build from latest qcom-next tag (default distro: trixie)
./build-kernel.sh --latest-tag

# Build from specific tag
./build-kernel.sh --tag qcom-next-6.12.0-20260210

# Build from branch tip
./build-kernel.sh --branch qcom-next --distro noble

# Build with explicit LOCALVERSION
./build-kernel.sh --tag qcom-next-6.12.0-20260210 --localversion qcom-next-20260210

# Build debug variant
./build-kernel.sh --latest-tag --localversion debug --profiles debug

# Use local kernel source (skip clone)
./build-kernel.sh --local-source /path/to/kernel-source --localversion qcom-next-20260210

# The script auto-detects debian/ from its own directory — no --debian-dir needed

# Enable optional config fragments for a Docker-capable build
./build-kernel.sh --latest-tag --enable-configs docker,systemd-boot,qcom-imsdk

# Enable configs for a QEMU test build
./build-kernel.sh --latest-tag --enable-configs qemu-boot,docker
```

### Manual build (without build-kernel.sh)

```bash
# 1. Set up kernel source
git clone git@github.com:qualcomm-linux/kernel.git kernel-source
cd kernel-source
git checkout qcom-next

# 2. Copy debian/ packaging
cp -r /path/to/pkg-linux-kernel/debian .

# 3. [Optional] Activate config fragments for this build
cp debian/config-available/docker.config       debian/config/
cp debian/config-available/systemd-boot.config debian/config/

# 4. Determine KVER
make ARCH=arm64 defconfig
sed -i 's/CONFIG_LOCALVERSION_AUTO=y/# CONFIG_LOCALVERSION_AUTO is not set/' .config
KVER=$(make -s ARCH=arm64 LOCALVERSION="-qcom-next-20260210" kernelrelease)
echo "KVER=$KVER"

# 5. Prepare packaging (single handoff to packaging infrastructure)
make -f debian/rules prepare KVER="$KVER"

# 6. Build
dpkg-buildpackage -us -uc -b
```

### Using docker-pkg-build

`docker_deb_build.py` runs the build inside a Docker container with `sbuild`:

```bash
python3 docker_deb_build.py \
  --source-dir /path/to/kernel-source \
  --output-dir /path/to/output \
  --distro noble
```

`build-kernel.sh` calls this automatically after running `prepare`.

## Packages produced

### `linux-image-<KVER>-qcom` — Kernel image

Installed paths:
- `/boot/vmlinuz-<KVER>` — compressed kernel image
- `/boot/config-<KVER>` — kernel `.config`
- `/lib/modules/<KVER>/` — stripped kernel modules
- `/usr/lib/linux-image-<KVER>/` — all DTBs (vendor subdirs preserved)
- `/lib/modules/<KVER>/build` → `/usr/src/linux-headers-<KVER>/` (symlink)
- `/lib/modules/<KVER>/source` → `/usr/src/linux-headers-<KVER>/` (symlink)

Virtual packages provided: `linux-image`, `linux-image-arm64`

Maintainer scripts:
- `preinst` — no-op; coexistence of multiple kernel versions is handled by
  dpkg automatically (each version has a unique package name)
- `postinst` — runs `depmod`, then `/etc/kernel/postinst.d/` hooks
  (triggers `update-initramfs` and `update-grub`)
- `postrm` — runs `/etc/kernel/postrm.d/` hooks (triggers `update-grub`)

### `linux-headers-<KVER>-qcom` — Kernel headers

Installed paths:
- `/usr/src/linux-headers-<KVER>/` — UAPI headers, `Makefile`, `.config`,
  `Module.symvers`, `scripts/`, `arch/arm64/` headers

Required for out-of-tree kernel module compilation (DKMS, etc.).

Virtual packages provided: `linux-headers`, `linux-headers-arm64`

### `linux-image-<KVER>-qcom-dbg` — Debug symbols

Installed paths:
- `/lib/modules/<KVER>/` — unstripped kernel modules (with debug symbols)
- `/usr/lib/debug/boot/vmlinux-<KVER>` — unstripped vmlinux

Depends on `linux-image-<KVER>-qcom` (same version).

Virtual packages provided: `linux-image-dbg`, `linux-image-arm64-dbg`

---

## The `build` and `source` symlinks

`/lib/modules/<KVER>/build` and `/lib/modules/<KVER>/source` are symlinks
pointing to `/usr/src/linux-headers-<KVER>/`. These are used by:
- `make -C /lib/modules/$(uname -r)/build` — standard out-of-tree module build
- DKMS — automatic module rebuild on kernel update
- Any tool that needs the kernel build infrastructure

**Why they are set explicitly:** `make modules_install` creates these symlinks
pointing to the build machine's source tree (an absolute path that does not
exist on the target system). The packaging explicitly removes and recreates
them to point to the installed headers location, matching the behaviour of
Debian's official `linux` package and `make bindeb-pkg`.

## Install and remove

```bash
# Install kernel image
sudo dpkg -i linux-image-<KVER>-qcom_1-1_arm64.deb

# Install headers (for out-of-tree module building)
sudo dpkg -i linux-headers-<KVER>-qcom_1-1_arm64.deb

# Install debug symbols
sudo dpkg -i linux-image-<KVER>-qcom-dbg_1-1_arm64.deb

# Remove a specific kernel version (does not affect other installed kernels)
sudo dpkg -r linux-image-<KVER>-qcom

# List all installed qcom kernels
dpkg -l 'linux-image-*-qcom'
```

Multiple kernel versions coexist safely — each has a unique package name.
Kernels are only removed on explicit `dpkg -r` or `apt autoremove`.

## Kernel configuration

### Config build order

`override_dh_auto_configure` in `debian/rules` applies kernel configuration in
this fixed order:

| Step | Source | Description |
|------|--------|-------------|
| 1 | `make defconfig` | ARM64 base defconfig from kernel source |
| 2 | `debian/rules` | Disable `CONFIG_LOCALVERSION_AUTO` (prevents git hash in `uname -r`) |
| 3 | Unified fragment pipeline | `arch/arm64/configs/qcom.config` (if present), then `debian/config/*.config` (sorted) |
| 4 | `debian/rules` | Re-check `CONFIG_LOCALVERSION_AUTO` (merges may re-enable it) |
| 5 | `arch/arm64/configs/debug.config` | Debug options — only when `DEB_BUILD_PROFILES=debug` |

### Unified config fragment pipeline (step 3)

All config fragments are processed through a single unified pipeline:

1. Collect fragments in order:
   - `arch/arm64/configs/qcom.config` first (kernel-source fragment, present in `qcom-next`)
   - then `debian/config/*.config` in sorted filename order
2. Append all collected fragments to `.config`
3. Run `make olddefconfig` once to resolve all conflicts and fill in dependencies

`qcom.config` and `debian/config/*.config` use exactly the same mechanism.

### `debian/config/` — always-applied fragments

Fragments in `debian/config/` are committed to the repo and applied on every
build. Add a fragment here for options that should always be present in the
Qualcomm Linux kernel package.

**Current always-applied fragments:**

| File | Purpose |
|------|---------|
| `debian/config/squashfs.config` | SQUASHFS options required for Ubuntu compatibility |

### `debian/config-available/` — optional fragment library

Fragments in `debian/config-available/` are committed to the repo but **not
applied automatically**. They are a library of well-known, tested config sets
that can be selectively activated per build.

**To activate a fragment**, copy it from `config-available/` into `config/`
before running the standard build flow. `build-kernel.sh` does this via
`--enable-configs`. CI pipelines do the same copy step before invoking
`dpkg-buildpackage`.

**Available fragments:**

| Fragment | Contents | Use case |
|----------|----------|----------|
| `docker.config` | Namespaces, cgroups, nftables, netfilter, overlay FS, VXLAN, AppArmor | Builds targeting Docker/container workloads |
| `systemd-boot.config` | `CONFIG_EFI_ZBOOT=y` | EFI systems using systemd-boot |
| `qcom-imsdk.config` | DMABUF heaps | Multimedia/camera pipelines (Qualcomm IMSDK) |
| `qemu-boot.config` | `SCSI_VIRTIO`, `DRM_VIRTIO_GPU` | QEMU test builds |
| `usb-can.config` | slcan, gs_usb, peak_usb | USB CAN adapter demos/testing |

**Activating via `build-kernel.sh`:**

```bash
# Enable docker + systemd-boot + qcom-imsdk for this build
./build-kernel.sh --latest-tag --enable-configs docker,systemd-boot,qcom-imsdk
```

**Activating manually (CI or manual build):**

```bash
# Copy desired fragments into config/ before running prepare
cp debian/config-available/docker.config       debian/config/
cp debian/config-available/systemd-boot.config debian/config/
cp debian/config-available/qcom-imsdk.config   debian/config/

# Then run the standard flow — no special flags needed
make -f debian/rules prepare KVER="$KVER"
dpkg-buildpackage -us -uc -b
```

This is the same operation whether run locally or in CI. CI pipelines simply
set which fragments to copy based on the pipeline's purpose (e.g., a
Docker-capable image build vs. a minimal QEMU test build).

**Adding a new fragment to the library:**

```bash
cat > debian/config-available/my-feature.config << 'EOF'
# My feature description
CONFIG_MY_FEATURE=y
CONFIG_MY_FEATURE_EXTRA=m
EOF
```

Commit it to `config-available/`. It will not affect any build until explicitly
activated via `--enable-configs my-feature` or a manual copy to `config/`.

### Debug build (step 5)

```bash
# Via build-kernel.sh
./build-kernel.sh --latest-tag --profiles debug

# Via dpkg-buildpackage directly
DEB_BUILD_PROFILES=debug dpkg-buildpackage -us -uc -b
```

Merges `arch/arm64/configs/debug.config` from the kernel source if present.

## Configuration knobs

| Knob | How to set | Effect |
|------|-----------|--------|
| `KVER` | `make -f debian/rules prepare KVER=...` | Sets all package names and install paths |
| `LOCALVERSION` | Encoded in `KVER` via `build-kernel.sh` | Sets `uname -r` suffix |
| Always-on config | Add `*.config` to `debian/config/` | Applied on every build |
| Optional config | `--enable-configs name` or copy from `debian/config-available/` | Activated per build |
| Out-of-tree build dir | `O=<dir>` or `KBUILD_OUTPUT=<dir>` | Reads artifacts from objdir |
| Debug build | `DEB_BUILD_PROFILES=debug` or `--profiles debug` | Merges `arch/arm64/configs/debug.config` |
| Parallel jobs | `DEB_BUILD_OPTIONS=parallel=N` | Controls `make -jN` |
